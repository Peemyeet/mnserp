# เกณฑ์ทำความสะอาด `customer` / `customer_contact` (db_mns)

เป้าหมาย: ให้ส่งออกรายชื่อผู้ติดต่อ B2B, แคมเปญอีเมล, และมอบหมาย key account ได้จริง โดยสอดคล้อง PDPA

## 1. ข้อมูลถือว่า “ไม่พร้อมใช้” (ต้องแก้ก่อนนำไปแคมเปญ)

| เงื่อนไข | ตาราง/คอลัมน์ | การแก้ |
|-----------|----------------|--------|
| เบอร์โทรเป็นเลขหลักเดียวหรือสั้นผิดปกติ (เช่น `1`, `2`, `02`) | `customer_contact.cs_tel` | ติดต่อลูกค้ายืนยันเบอร์จริง หรือลบแถวทดสอบ |
| อีเมลว่าง | `customer_contact.cs_email`, `customer.contact_email` | ขอจากฝ่ายจัดซื้อ / ใบเสนอราคา / ลายเซ็นอีเมลเดิม |
| อีเมลหลายค่าในช่องเดียว (คั่นด้วย `/` หรือช่องว่าง) | `customer_contact.cs_email` | แยกเป็นแถว `customer_contact` คนละแถวต่อหนึ่งอีเมล หรือเก็บอีเมลหลักแถวเดียว |
| ชื่อผู้ติดต่อเป็นคำทั่วไป (เช่น `คุณ` เท่านั้น) | `customer_contact.cs_name` | ขอชื่อ-นามสกุลหรือตำแหน่ง |

## 2. Key accounts — เกณฑ์เติมอีเมลฝ่ายจัดซื้อ

กำหนด key account จากข้อมูลจริง (อย่างใดอย่างหนึ่งหรือผสม):

- จำนวน `job_data` ต่อ `customer_id` อยู่ในเปอร์เซ็นไตล์บนของบริษัท (เช่น top 10%)
- ยอดรวม `bill_quota.grand_total` ต่อลูกค้าอยู่ในกลุ่ม top ตามนโยบายการเงิน
- ลูกค้าที่มี `sale_id` ชี้มาที่พนักงานขายคนหลัก (ถ้ามีการตกลงบัญชีลูกค้าแล้ว)

**ขั้นตอน:**

1. รัน query สรุปใน `database/queries/customer_segment_summary.sql` เพื่อจัดอันดับลูกค้า
2. สำหรับแต่ละรายในกลุ่ม key: ต้องมีอย่างน้อย **หนึ่ง** `customer_contact` ที่มี `cs_email` ถูกต้องและเป็นผู้มีอำนาจจัดซื้อหรือผู้ประสานหลัก
3. ถ้ามีเฉพาะเบอร์โทร: บันทึกใน `cs_tel` และตั้ง flag ภายใน (หรือหมายเหตุใน CRM) ว่า “ยังไม่มี consent อีเมล”

## 3. PDPA และการใช้ข้อมูล

- ใช้ข้อมูลติดต่อเพื่อวัตถุประสงค์ที่แจ้ง (เช่น ส่งข่าวบริการซ่อม / ใบเสนอราคา) และเก็บหลักฐานความยินยอมถ้ามีการสมัครรับข่าว
- ไม่รวมข้อมูลจาก `market` (ซัพพลายเออร์) เข้ากับรายชื่อลูกค้าโดยไม่แยกวัตถุประสงค์ — เป็นคนละความสัมพันธ์ทางธุรกิจ

## 4. ความสม่ำเสมอของรหัสลูกค้า

- ผูกทุก contact ด้วย `cus_code` ให้ตรงกับ `customer.cus_code`
- หากพบ `cus_code` ซ้ำกันสองความหมาย (ข้อมูลจริงในระบบเก่า): แยกบัญชีในระบบใหม่หรือทำ master data cleanup ก่อนรันแคมเปญ
